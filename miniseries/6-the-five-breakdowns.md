# Episode 6 — The Five Breakdowns (A Lencioni-Style Readout)

Dax called the team into the smallest conference room, the one with the blunt window and the reliable speakerphone. He shut the door and put his laptop on the table, as if the weight would hold the day in place.

"We just ran CinderCloud," he said. "Now I need a leader memo. Lencioni-style. Five breakdowns, each with a move leaders can run. Two pages. No slide deck."

Myles crossed his arms. "You want diagnosis, not inspiration."

"Exactly," Dax said. "Concrete failure modes. Symptoms. Root constraints. Commands that fix them. And evidence. I don't want to be sold."

Rina glanced at the clock. "They had ninety minutes before the partner sync."

"Then we run the system," Jonah said. He opened a shared doc, the cursor blinking like a metronome.

Myles wrote the first header where everyone could see it.

## /problem-space

Rina called it out as he typed.

**Optimization target:** a memo leaders can act on in one meeting.  
**Constraints treated as real:** two pages, no new jargon, no org rewrite, no heroics required.  
**Constraints we should question:** "leaders already know this," "busy equals progress."

Kieran leaned back. "Let's not smuggle in a manifesto."

"Agreed," Rina says. "It has to read like a diagnosis."

Myles kept going.

## /problem-statement

Old: "What did we learn across five engagements?"  
New: "How can a leader spot breakdowns early and apply the smallest corrective command?"

Dax nods once. "That's the memo."

Rina tapped the table. "Aim it."

## /aim

They write it together.

**Aim:** A leader can use this memo to identify a live breakdown in their org and choose a corrective move within a week.

Jonah drew four boxes on the whiteboard and labeled them without looking up.

## /solution-space

**Tarp:** list commands with definitions.  
**Nearest Peak:** list breakdowns without evidence.  
**Beyond:** tie each breakdown to a scene and a corrective command.  
**Terraform:** end with reusable artifacts leaders can lift.

"Beyond plus Terraform," Dax said. "We show the scenes and ship the artifacts."

"Then we /execute the memo like we do everything else," Myles said. "Three passes."

## /execute

Pre-flight: outline the five breakdowns and required commands.  
Build: add evidence from episodes and root constraints.  
Detect drift: if we start preaching, stop.  
Salvage: pull the sharpest evidence and delete the rest.

Kieran squinted. "We're going to preach. Someone /dissent us."

Rina drew a line and wrote the command in the doc.

## /dissent

What would make this wrong?
- If it turns into doctrine instead of diagnosis.
- If leaders can't see themselves in it.
- If it gives them more meetings instead of fewer.

Jonah pointed at the last bullet. "That's the trap. Ticket thinking survives because it's emotionally safe."

Dax heard the line and repeated it, softer. "So we make the moves safe, not the avoidance."

Myles flipped to a clean page. "Before we write, we salvage evidence. One moment per breakdown."

## /salvage

They pulled notes from each engagement:
- Harborview's scanner taxonomy and the first /problem-space.  
- KiteRail's tool debate and the /problem-statement rewrite.  
- SablePay's manifest aim that turned a doc into a default.  
- Aurum's guardrail set and the drift it revealed.  
- CinderCloud's distill loop and retrieval-at-need.

Rina skimmed the outline and ran a fast check.

## /review

"Does this match the aim?" she asks. "Can a leader act in a week, or did we make it a thesis?"

Dax closed his laptop. "We ship at four. Short lines. Clear commands."

He stood, already moving.

## /ship

"Send it to the partners as a memo, not a deck," he says. "If they forward it, it should still work."

They wrote. Then they cut. Then they wrote again.

## Internal Readout — Dax Mercer (Northstar)

**Purpose**
This is the internal readout after CinderCloud. It is not a transcript. It is a diagnosis leaders can act on.

Two anchor lines guided the framing:
- "Ticket thinking survives because it's emotionally safe."  
- "Leadership quality now hinges on how well we search, not how well we craft."

Below are five breakdowns we saw across Harborview, KiteRail, SablePay, Aurum, and CinderCloud. Each includes symptoms, root constraint, corrective commands, and evidence.

### Breakdown 1: No Shared Problem Space (Everyone Optimizes Something Different)

**Symptoms**
- Smart people disagree constantly, but can't explain why.
- Work looks busy but doesn't converge.
- Agents produce impressive outputs that land on the wrong target.

**Root constraint**
The team never writes down what constraints are real, so they optimize for different physics.

**Corrective commands**
- Run /problem-space in writing, not as hallway alignment.
- Use /dissent to surface which constraints might be wrong.

**Evidence**
Harborview normalized scanner failures as "just how it is." The room shifted only after Rina and Myles wrote /problem-space and named manual approvals as a questionable constraint. Until then, each role optimized its own pain.

**Leader move**
Require a written /problem-space before work starts. Treat it like a safety checklist: no map, no move.

### Breakdown 2: The Problem Statement Never Changes (So the Solution Space Stays Small)

**Symptoms**
- Teams keep applying tarps to the same leak.
- Tool debates substitute for outcome debates.
- "We're waiting for X tool" becomes the plan.

**Root constraint**
Framing is treated as fixed, so the solution space stays trapped in the nearest peak.

**Corrective commands**
- Force a /problem-statement rewrite before deep execution.
- Run /solution-space explicitly: Tarp -> Nearest Peak -> Beyond -> Terraform.

**Evidence**
At KiteRail, the room was drifting toward a vendor fight until the /problem-statement changed from "make agents code faster" to "become collectively right." That opened the search for a shared loop instead of a shared tool.

**Leader move**
Make problem statement rewrites a habit. If the statement never changes, you never search; you just polish.

### Breakdown 3: No Aim (So Activity Looks Like Progress)

**Symptoms**
- "Shipped" doesn't change behavior.
- Metrics move; users don't.
- The team can't tell if the work worked.

**Root constraint**
The outcome is not defined as behavior change, so verification is ambiguous.

**Corrective commands**
- Write /aim as a behavioral outcome with a feedback signal.
- Run /review against the aim, not the artifact.

**Evidence**
At SablePay, the manifest was just another doc until Myles wrote /aim as behavior change: "a junior can start a session with the right context without a senior in the loop." The work became measurable the moment the aim did.

**Leader move**
Every initiative starts with a one-sentence /aim and the "how we'll know" signal. Make it visible where work begins.

### Breakdown 4: Ungoverned Execution (The Chaos Dragon Eats the Sprint)

**Symptoms**
- Endless drafts, reversals, and "almost done" work.
- Humans babysit agents instead of steering a system.
- Rework grows while "done" gets fuzzier.

**Root constraint**
Execution is not governed by drift detection and salvage, so sunk cost takes over.

**Corrective commands**
- Use /execute as a loop: pre-flight -> build -> detect drift -> salvage early.
- When drift is detected, /salvage before continuing.

**Evidence**
At Aurum, an agent produced a clean-looking diff that failed in the lab. The team only regained control when they treated /execute as a loop and wrote salvage criteria before pushing further.

**Leader move**
Define drift signals and kill criteria before the work begins. If the team can't say when to stop, they won't.

### Breakdown 5: No Terraforming (Learning Doesn't Compound)

**Symptoms**
- The same failures recur across teams and quarters.
- Standards exist as preferences, not enforceable constraints.
- Local excellence doesn't scale; everything relies on heroes.

**Root constraint**
Learning is captured as prose, not as environment changes (guardrails, defaults, delivery-path improvements).

**Corrective commands**
- After each meaningful learning, decide what should become impossible next time.
- Use /ship to reduce delivery-path tax so learning reaches users fast.
- Use /review to prevent guardrails from ossifying; apply sunset clauses.

**Evidence**
At CinderCloud, the team threw away a month of "best practice" docs and replaced them with a smaller system: distills, guardrails, a manifest template, and retrieval-at-need. That was the first time learning stopped leaking across teams.

**Leader move**
Terraform one thing after every incident or major win. If nothing in the environment changes, the org will forget by default.

## The Minimal Playbook (Three Moves)

This is the smallest reusable sequence we want leaders to run.

1) Start with /problem-space + /aim (write it down).
2) Treat /review + /dissent as a safety system, not bureaucracy.
3) When drift appears, /salvage immediately and terraform one thing before you restart.

## If You Only Measure Three Things

These metrics tell you whether the loop is working.

1) **Reversal rate:** how often work is undone because it missed the aim.
2) **Time-to-first-correct-change:** how long it takes to land the first change that actually matches the aim.
3) **Repeat failure recurrence:** how often the same failure mode shows up across teams or quarters.
