I can sit with a team, map the people, incentives, and constraints around a workflow, then build the code that has to run inside it. When the code exposes the wrong problem, I change the plan.

I do that with teams adding agents to workflows. I also build public tools and methods, run experiments, and write.

I spent more than fifteen years building web applications, coaching developers, and leading engineering teams. Then I focused on data and led data organizations at JPMC, Zapier, and Shopify.

After NerdWallet, I assumed the next move was another engineering or data leadership role. Hands-on development answered the question underneath that plan: yes, I can still do this. The more useful question now is where I want to invest my time and judgment.

Leading those organizations changed what I look for when I build. A technically plausible system can fail because nobody owns the decision, the team cannot replay a bad result, or the metric rewards usage instead of effectiveness. I look for those failures before treating the code as finished.
